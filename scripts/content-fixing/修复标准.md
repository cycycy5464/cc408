# 真题修复标准

> 所有真题文件必须遵守的格式化规范。
> 任何修复操作完成后，对照此清单逐项检查。

---

## 一、文件结构

```

```

| 部分 | 说明 | 示例 |
|------|------|------|
| `{年份}` | 4 位数字 | `2011` |
| `{科目缩写}` | `ds` / `co` / `os` / `cn` | `ds` |
| `{题号三位}` | 001~047 | `041` |

| 选择题 | 012~022 | 计算机组成原理 | 11 |
| 选择题 | 023~030 | 操作系统 | 8 |

### 1.3 Frontmatter

```yaml

  - "2011"

- `number` 与文件名中的题号一致

## 二、选择题格式

### 2.1 选项

**正确**（4 项连续，无空行）：
```markdown
A\. 选项一
B\. 选项二
C\. 选项三
D\. 选项四
```

**错误**（有空行 → Goldmark 拆分为独立 `<p>`）：
```markdown
A\. 选项一

B\. 选项二   ← 这里不能有空行
```

**错误**（纯空格分隔 → JS 无法解析）：
```markdown
A     B     C     D    ← 必须有 `.` 分隔符
```

### 2.2 答案标记

```markdown
[tag_link]

正确答案：A

解析内容...
```

**规则**：
- `[tag_link]` 与 `正确答案` 之间必须空一行（否则合并为同一 `<p>`）
- `正确答案` 与解析之间必须空一行（否则解析文本随 `<p>` 一并删除）
- `[tag_link]` 前不能有其他文字（独占一行）

### 2.3 解析

解析内容不能为空。解析中的 `【解析】` 标记由 JS 自动替换为样式标题，无需手动处理。

### 2.4 选项与 [tag_link] 的顺序

```markdown
A\. 选项一
B\. 选项二
C\. 选项三
D\. 选项四

[tag_link]

正确答案：A

解析内容...
```

选项必须在 `[tag_link]` 之前（否则 JS 会将选项当作解析移入面板）。

### 2.5 选项与 [tag_link] 粘连

**严禁** D 选项与 `[tag_link]` 在同一行：

```markdown
D\. n-1, n-1
[tag_link]     ← 错误！D 选项被吸入 `<p>` 后消失
```

**必须** D 选项与 `[tag_link]` 之间有且仅有一个空行：

```markdown
D\. n-1, n-1

[tag_link]     ← 正确
```

**根本原因**：Goldmark 将没有空行分隔的行合并为同一个 `<p>` 段落，JS 提取选项时把 `[tag_link]` 文本当作 D 选项尾部处理，导致 D 选项在前端不显示。

### 2.6 选择题去重

从爬虫数据覆盖选项后，**必须清理旧选项行**，确保最终只有 4 行 `A\.xxx` ~ `D\.xxx`。

**可能出现重复的情况**：
- 修复脚本追加了新格式 `A.xxx`（不带反斜杠），原有 `A\.xxx` 未被删除 → 8 个选项行
- KaTeX HTML 格式被作为选项行保留 → 内容为空或 HTML 标签，需要替换为纯文本

**修复方案**：
1. 优先保留 `A\` 格式（带反斜杠），删除 `A.` 格式（纯点）的重复行
2. 若原始 `A\` 格式破损（一行多项/空内容），则将完整的 `A.` 行转为 `A\.`
3. 若 KaTeX 选项提取为空，回退为纯文本格式

### 2.7 选择题代码块包裹

题干中的代码段必须用 ```c 包裹，否则 Goldmark 将代码暴露为普通段落，格式混乱。

```markdown
下列程序段的时间复杂度是（）。

```c
count = 0;
for (k = 1; k <= n; k *= 2)
    for (j = 1; j <= n; j++)
        count++;
```

A\.O(log₂n)
B\.O(n)
C\.O(nlog₂n)
D\.O(n²)
```

**修复方案**：将裸露的 `k*=2`、`j++` 等代码行包裹在 ```c 和 ``` 之间。

### 2.8 选择题前端 A 选项不显示

爬虫数据以 `A\xxx`（反斜杠后紧跟内容，无点）格式追加到文件中。但前端 JS 提取选项的正则表达式需要 `A\` 或 `A.` 格式才能识别。

**修复方案**：确保选项格式统一为 `A\.xxx`（反斜杠+点+空格+内容），爬虫数据中的格式需要转换。

## 三、综合题格式

### 3.1 结构

```markdown
题干内容...

[知识标签](/study_methods/tags/...)

[tag_link]

1）第一问解析...

2）第二问解析...
```

### 3.2 [tag_link]

- 每道综合题 **必须有且仅有 1 个** `[tag_link]`
- 位于最后一个知识标签链接之后、解析内容之前
- 与解析内容之间**空一行**

### 3.3 子问题分段

子问题（1）、2）、3）等）之间用空行分开，确保 Goldmark 渲染为独立 `<p>`：

```markdown
1）第一问解析...

2）第二问解析...
```

### 3.4 代码块

**核心规则**：已有 ` ```c ` 的绝不再加第二个。仅对无语言标签的 ` ``` ` 且内容为 C 代码的才添加 `c`。

C 代码判据（行内出现任一即算）：
```
int / char / float / double / void / struct / typedef
return / if / else / for / while / do / switch / case / break / continue
// 注释 / /* / #include
printf / scanf / malloc / free / sizeof
load / store / inc / dec / add / sub / push / pop  (汇编类)
```

### 3.5 表格

表格前后必须有空行：

```markdown
计算结果如下。

| 列1 | 列2 |
| --- | --- |
| 数据 | 数据 |

由以上可知...
```

### 3.6 表格内部结构

**表格表头行与分隔行之间不能有空行**：

✅ 正确：
```markdown
| 目的网络 | 下一跳 | 接口 |
| --- | --- | --- |
| 169.96.40.0/23 | 176.1.1.1 | S1 |
```

❌ 错误（表头和分隔行之间有空行 → Goldmark 不识别为表格）：
```markdown
| 目的网络 | 下一跳 | 接口 |

| --- | --- | --- |
| 169.96.40.0/23 | 176.1.1.1 | S1 |
```

### 3.7 零宽空格清理

爬虫数据中会混入零宽空格 `\u200b`，影响中文排版，**必须替换为普通空格**。

常见出现位置：
- 邻接矩阵、ASCII 图的数字/字符之间
- 代码段的排版空隙
- 解析中的英文单词间隔

修复方案：
```python
text = text.replace('\u200b', ' ')
```

## 四、科目命名一致性

### 4.1 `subjects` 字段

所有文件的 `subjects` 字段**必须使用全称**：

| 科目 | 正确 | 错误 |
|------|------|------|
| 计算机组成原理 | `"计算机组成原理"` | `"组成原理"` |
| 其他科 | `"数据结构"` / `"操作系统"` / `"计算机网络"` | —（无简称） |

`subject: "组成原理"` 的文件在前端会与 `subject: "计算机组成原理"` 的被归为不同分组，导致同一道选择题在前端重复出现（显示为 8 个选项）。

### 4.2 冗余文件

若某题号同时存在 `co-{n}.md` 和 `ds-{n}.md`（n ∈ [012, 022]），且 `ds-` 文件 subject 为 `"组成原理"`：
- 此为爬虫数据产生的冗余文件
- **优先保留 `co-` 文件**（subject 全称准确）
- **直接删除 `ds-` 文件**（不做 git restore）

`content/exam/408quiz/{year}/content.md` 是爬虫遗留的旧版整卷文件。若存在且与单题渲染冲突，应删除。

## 五、禁止操作

| 操作 | 原因 |
|------|------|
| `git restore` / `git checkout` 恢复单题 | 会覆盖人工修复的格式和内容 |
| `git push --force` | 丢失历史 |
| 批量全局正则替换脚本 | 之前 `c3c8178` 提交因此误删选项行 |
| 直接修改模板正则 | 选项分隔符标准为 `[A-D]\s*\.`，源文件适配而非修改模板 |

## 六、换行符

**所有 `.md` 文件必须使用 Unix 换行符（`\n`）**。

Windows 的 `\r\n` 和爬虫源的 `\r\r\n`（双回车）会在 GitHub Actions 的 Linux 构建中触发 Hugo panic：

```
panic: runtime error: slice bounds out of range
```

**修复**：
```python
raw.replace(b'\r\r\n', b'\n').replace(b'\r\n', b'\n')
```
